# BaroName Product Design

작성일: 2026-06-21

## 1. 이름 선택

선택 이름: **바로네임 / BaroName**

선택 이유:

- 한국어 사용자에게 “필요한 이름을 바로 만든다”는 효용이 즉시 전달된다.
- `NameCraft`는 세련되지만 흔한 글로벌 SaaS 느낌이 강하고, `AI NameLab`은 실험실 인상이 있어 초보 사용자에게 약간 차갑다.
- `바로네임`은 비개발자, 1인 창업자, 크리에이터가 처음 들어와도 용도를 바로 이해한다.
- 영문 표기 `BaroName`은 도메인, 앱명, 로고 확장에도 무리가 없다.

제품 한 줄 정의:

> 브랜드 전문가의 사고방식을 AI와 인터랙션으로 풀어낸, 이름 후보 생성·비교·선별 작업대.

핵심 약속:

> 무작위 이름 생성기가 아니라, 사용자가 자기 브랜드의 방향을 잡아가며 좋은 후보를 좁혀가는 네이밍 빌더.

## 2. 제품 방향

BaroName은 “이름 생성”보다 “이름 결정”을 돕는 서비스여야 한다. 많은 AI 네이밍 도구의 문제는 후보를 많이 뽑아주지만, 왜 좋은지, 어떤 기준으로 골라야 하는지, 다음에 무엇을 수정해야 하는지 알려주지 않는다는 점이다.

따라서 BaroName의 중심 가치는 다음 세 가지다.

1. **전문가식 발상 프레임**
   네이밍 기법, 어감, 기억성, 차별성, 확장성 기준을 결과 카드에 함께 보여준다.

2. **사용자 취향 학습**
   사용자가 저장한 후보를 기반으로 다음 생성 방향을 조정한다. 실제 모델 파인튜닝이 아니라 세션 내 선호 요약을 프롬프트에 반영하는 방식으로 구현한다.

3. **무료 API 친화 설계**
   Gemini 무료 티어를 전제로, 새로고침성 호출을 줄이고 한 번의 호출에서 구조화된 결과를 충분히 받는다.

## 3. 대상 사용자

주 사용자:

- 1인 창업자, 사이드 프로젝트 제작자
- 유튜브, 숏폼, 뉴스레터, 커뮤니티 채널 운영자
- 앱, 웹서비스, 스토어, 제품명을 정해야 하는 비개발자
- 초기 브랜딩 비용을 쓰기 어려운 소규모 팀

사용자 상황:

- “괜찮은 이름이 생각나지 않는다.”
- “이미 떠오른 이름이 좋은지 모르겠다.”
- “한글/영문 이름을 같이 보고 싶다.”
- “도메인, SNS, 상표 같은 체크리스트가 막막하다.”

## 4. 핵심 컨셉

초안의 3단 스플릿 뷰는 유지하되, 단순 좌측 설정·중앙 결과·우측 저장 구조에서 한 단계 더 발전시킨다.

추천 구조:

- 좌측: **브랜드 브리프**
- 중앙: **네이밍 스튜디오**
- 우측: **픽 보드 & 판정표**

이름 생성은 한 번의 버튼으로 끝나지 않고 다음 흐름으로 진행된다.

1. 방향 입력
2. 후보 생성
3. 마음에 드는 후보 저장
4. 저장 후보 기반으로 유사/반대/더 과감한 후보 재생성
5. 최종 후보 비교
6. 체크리스트와 내보내기

## 5. 네이밍 엔진

초안의 네 가지 기법에 더해, 실제 사용자가 더 다양한 방향을 탐색할 수 있도록 8개 엔진으로 확장한다.

### 5.1 Descriptive

서비스/제품의 정체를 직관적으로 드러내는 이름.

예: 바로배송, 모두의지도, SimpleNote

장점:

- 이해가 빠르다.
- 초기 마케팅 비용이 낮다.

주의:

- 흔해 보일 수 있다.
- 도메인이나 상표 충돌 가능성이 높다.

### 5.2 Blended

두 단어를 합치거나 줄여 새로운 단어처럼 만드는 방식.

예: Instagram, Microsoft, 핀다

장점:

- 의미와 고유성을 동시에 확보하기 쉽다.

주의:

- 억지 조합은 촌스럽게 보인다.

### 5.3 Metaphorical

브랜드의 감정, 세계관, 결과 상태를 상징으로 표현한다.

예: Apple, Amazon, Toss

장점:

- 스토리와 확장성이 좋다.

주의:

- 초기에는 설명이 필요할 수 있다.

### 5.4 Invented

완전히 새로운 조어를 만든다.

예: Kodak, Xerox, Spotify

장점:

- 고유성이 높고 글로벌 확장이 쉽다.

주의:

- 발음성과 기억성을 강하게 검증해야 한다.

### 5.5 Benefit-led

사용자가 얻는 이익을 이름에 담는다.

예: 빠른정산, BetterSleep

장점:

- 전환율 중심 서비스에 강하다.

주의:

- 기능명처럼 보일 수 있다.

### 5.6 Persona-led

타깃 사용자의 정체성이나 소속감을 이름화한다.

예: MakerLog, Momento, 직장인탐구생활

장점:

- 커뮤니티, 콘텐츠, 교육 서비스에 좋다.

주의:

- 타깃이 바뀌면 이름이 좁아질 수 있다.

### 5.7 Phonetic

발음, 리듬, 반복음, 받침감, 음절감을 중심으로 만든다.

예: 카카오, 쿠쿠, TikTok

장점:

- 기억성과 구전성이 좋다.

주의:

- 의미가 약하면 가볍게 느껴질 수 있다.

### 5.8 Founder/Story

창업자의 배경, 지역, 시작 계기, 철학에서 이름을 뽑는다.

장점:

- 진정성과 스토리텔링에 강하다.

주의:

- 사용자가 충분한 맥락을 입력해야 한다.

## 6. 입력 설계

좌측 패널은 “설문지”가 아니라 “브랜드 브리프 작성기”처럼 느껴져야 한다.

### 6.1 필수 입력

- 만들 이름의 대상: 앱, 웹서비스, 회사, 제품, 매장, 유튜브/채널, 뉴스레터, 커뮤니티, 기타
- 핵심 설명: 한 문장으로 무엇을 하는지
- 핵심 키워드: 최대 5개
- 원하는 언어: 한글, 영어, 한글+영어, 자유 혼합
- 선호 톤: 신뢰감, 모던함, 친근함, 프리미엄, 위트, 기술적, 감성적, 과감함

### 6.2 선택 입력

- 타깃 사용자
- 피하고 싶은 느낌
- 금지어
- 반드시 포함하고 싶은 단어
- 글자 수 범위
- 발음 난이도
- 국내 중심 / 글로벌 가능성
- 도메인 선호: `.com`, `.co.kr`, `.io`, `.app`, 무관

### 6.3 크리에이티브 슬라이더

슬라이더 이름: **창의성 강도**

5단계:

1. 명확함: 직관적이고 설명적인 이름
2. 균형: 이해하기 쉽지만 약간의 개성 포함
3. 신선함: 합성어와 은유를 적극 활용
4. 과감함: 조어, 낯선 어감, 라틴어/어근 활용
5. 실험적: 독특한 발음, 추상적 세계관 중심

중요 UX:

- 슬라이더 옆에 긴 설명을 붙이지 않는다.
- 단계 이름과 작은 예시만 보여준다.
- 생성 후 결과 카드에 “왜 이 창의성 단계에 맞는지”를 짧게 표시한다.

## 7. 생성 결과 카드

중앙 결과 카드는 보기 좋게 많은 정보를 담되, 첫눈에는 이름이 가장 강하게 보여야 한다.

카드 구성:

- 후보 이름
- 한 줄 포지셔닝
- 사용 기법
- 언어 유형
- 점수 4개: 기억성, 발음성, 차별성, 확장성
- 설명: 왜 이 이름이 어울리는지
- 주의: 혼동 가능성, 흔한 표현, 발음 이슈
- 액션: 저장, 비슷하게 더 만들기, 더 짧게, 더 프리미엄하게, 금지어 추가

점수는 AI가 절대 판정하는 것이 아니라 “초기 검토용 신호”로 표현한다.

추천 카드 예시:

- 이름: `Namo`
- 한 줄: 작은 브랜드를 빠르게 이름 붙이는 감각적인 조어
- 기법: Invented + Phonetic
- 점수: 기억성 86 / 발음성 92 / 차별성 74 / 확장성 80
- 주의: 기존 서비스명과 유사 가능성 수동 확인 필요

## 8. 픽 보드 & 비교

우측 패널은 단순 저장소가 아니라 의사결정 공간이어야 한다.

### 8.1 임시 픽

- 최대 7개까지 저장
- 카드에서 별 아이콘 클릭 또는 드래그로 추가
- 저장 시 “선호 신호”가 갱신된다.

선호 신호 예:

- 짧은 이름 선호
- 받침 없는 한글 선호
- 영어 2음절 선호
- 설명형보다 조어형 선호
- 부드러운 어감 선호

### 8.2 비교 테이블

최대 5개 후보 비교:

- 이름
- 글자 수 / 음절 수
- 발음 난이도
- 기억성
- 차별성
- 국내 적합성
- 글로벌 적합성
- 확장 가능 카테고리
- 리스크 메모

### 8.3 최종 후보 판정

최종 후보를 누르면 다음 질문으로 압축한다.

- 3초 안에 읽히는가?
- 한번 듣고 다시 말할 수 있는가?
- 카테고리가 바뀌어도 버틸 수 있는가?
- 부정적 의미나 발음 오해가 있는가?
- 도메인/SNS/상표 확인이 필요한가?

## 9. AI 설계

Gemini 무료 티어 사용을 전제로 한다.

공식 문서 기준으로 Gemini API의 한도는 RPM, TPM, RPD 같은 차원에서 관리되며, 실제 한도는 모델과 프로젝트 상태에 따라 AI Studio에서 확인해야 한다. 또한 무료 티어 입력/출력은 무료로 제공되지만, 무료 티어 데이터는 제품 개선에 사용될 수 있다. 따라서 사용자가 민감한 미공개 브랜드 전략이나 개인정보를 넣지 않도록 안내해야 한다.

참고:

- Gemini API rate limits: https://ai.google.dev/gemini-api/docs/rate-limits
- Gemini API pricing: https://ai.google.dev/gemini-api/docs/pricing
- Gemini structured outputs: https://ai.google.dev/gemini-api/docs/structured-output

### 9.1 권장 모델 운용

초기 권장:

- 기본 생성: `gemini-3.5-flash` 또는 당시 무료 티어에서 안정적으로 제공되는 Flash 계열
- 구조화 응답: JSON Schema 사용
- 긴 설명보다 후보 품질과 점수 데이터를 구조화해서 받기

모델명과 무료 한도는 변동될 수 있으므로 구현 직전 다시 확인한다.

### 9.2 호출 전략

무료 티어를 아끼기 위한 원칙:

- 한 번 생성 시 후보 12개를 한 번에 받는다.
- 후보별 설명은 1~2문장으로 제한한다.
- 재생성은 전체 재생성보다 “선택 후보 기반 변주”를 우선한다.
- 같은 입력값과 설정값은 캐시한다.
- 빠른 필터링, 정렬, 비교는 클라이언트/서버 로직으로 처리한다.
- 도메인/상표 확인은 자동 대량 조회가 아니라 최종 후보 단계에서만 수행한다.

### 9.3 API 호출 제한 UX

제한은 차갑게 막는 대신 제품 경험 안에 자연스럽게 녹인다.

- 생성 버튼 클릭 후 15~30초 쿨다운
- 하루 생성 횟수 표시: “오늘 남은 생성 8회”
- 같은 입력 재생성 시 “기존 후보를 먼저 변형해볼까요?” 제안
- 호출 실패 시 저장 후보와 로컬 변주 기능 제공
- 비로그인 사용자는 낮은 한도, 로그인 사용자는 프로젝트 저장과 함께 높은 한도

### 9.4 프롬프트 파이프라인

한 번의 생성은 네 단계 논리를 포함한다.

1. 입력 요약
   사용자의 브리프를 브랜드 전략 문장으로 압축한다.

2. 네이밍 방향 선택
   사용자가 선택한 엔진과 창의성 강도에 맞춰 생성 분포를 정한다.

3. 후보 생성
   이름, 설명, 기법, 언어, 점수, 리스크를 JSON으로 생성한다.

4. 자체 검토
   금지어, 길이, 중복, 발음 난이도, 지나치게 흔한 단어를 걸러낸다.

### 9.5 구조화 응답 필드

후보 데이터 예시:

```json
{
  "names": [
    {
      "name": "BaroName",
      "language": "mixed",
      "technique": ["descriptive", "blended"],
      "positioning": "필요한 순간 바로 쓰는 네이밍 작업대",
      "rationale": "한국어의 즉시성과 영문 SaaS 감각을 함께 준다.",
      "scores": {
        "memorability": 86,
        "pronunciation": 92,
        "distinctiveness": 72,
        "scalability": 80
      },
      "risks": ["일반 표현에 가까워 상표 확인이 필요함"],
      "variants": ["Baro Naming", "BaroLab", "NameBaro"]
    }
  ],
  "sessionInsight": "사용자는 짧고 직관적인 한영 혼합 이름을 선호할 가능성이 높음"
}
```

## 10. 정보 구조

### 10.1 첫 화면

첫 화면은 랜딩 페이지가 아니라 바로 작업 가능한 앱 화면이다.

상단:

- 좌측: BaroName 로고
- 중앙: 프로젝트명 입력
- 우측: 생성 횟수, 저장, 내보내기, 설정

본문:

- 3단 스플릿 뷰

하단:

- 생성 상태, 마지막 저장 시간, API 상태

### 10.2 좌측 패널

구성 순서:

1. 대상 유형
2. 한 문장 설명
3. 키워드
4. 톤 선택
5. 창의성 강도
6. 언어
7. 네이밍 엔진
8. 금지어
9. 생성 버튼

좌측 패널은 접을 수 있어야 한다. 모바일에서는 상단 시트로 열린다.

### 10.3 중앙 피드

상태:

- 빈 상태: 샘플 브리프 카드 3개
- 생성 중: 글자 조합 애니메이션
- 결과 있음: 후보 카드 리스트
- 오류: 재시도, 기존 후보 보기

카드 정렬:

- 추천순
- 짧은 이름순
- 기억성 높은 순
- 조어 중심
- 설명형 중심

### 10.4 우측 패널

구성:

- 비교 버튼
- 프로젝트 저장 목록

## 11. UI 스타일

방향:

- Apple 느낌의 정돈된 생산성 앱
- 과한 그라데이션, 장식형 카드, 마케팅식 히어로 지양
- 이름 후보가 가장 돋보이는 조용한 인터페이스

색상:

- 배경: 부드러운 화이트/쿨 그레이
- 주요 텍스트: 거의 검정에 가까운 차콜
- 포인트: 신뢰감 있는 블루 또는 민트 계열을 소량
- 경고/주의: 차분한 앰버
- 저장/선택 상태: 선명한 블루 또는 그린

타이포그래피:

- 한글 가독성 좋은 Pretendard 계열
- 후보 이름은 24~32px
- 카드 설명은 13~15px
- 버튼 텍스트는 짧고 명확하게

컴포넌트:

- 카드 반경은 8px 이하
- 버튼은 아이콘 중심으로 구성하고 툴팁 제공
- 체크박스보다 네이밍 엔진은 토글 칩 형태가 적합
- 창의성 강도는 슬라이더와 단계 라벨 사용
- 비교는 테이블보다 압축된 매트릭스 형태가 적합

## 12. UX 디테일

### 12.1 생성 중 경험

초안의 “글자가 조합되다 멈추는 애니메이션”은 좋다. 다만 너무 장난스럽지 않게, 이름 후보 카드의 빈 자리에서 짧게만 사용한다.

문구 예:

- 방향을 정리하는 중
- 어감을 다듬는 중
- 겹치는 표현을 걷어내는 중
- 후보를 점수화하는 중

### 12.2 저장 피드백

후보 저장 시:

- 카드가 우측으로 부드럽게 이동
- 우측 보드에 바로 추가
- “짧고 발음 쉬운 이름을 선호하는 것 같아요” 같은 세션 인사이트 표시

### 12.3 재생성 방식

각 카드에서 바로 가능한 액션:

- 비슷하게
- 더 짧게
- 더 고급스럽게
- 더 친근하게
- 한글화
- 영문화
- 이 단어는 제외

이 방식은 무분별한 전체 재생성을 줄이고, 사용자가 “방향을 잡고 있다”는 감각을 준다.

## 13. 추천 추가 기능

### 13.1 네이밍 점수판

이유:

- 초기부터 태그라인까지 많이 만들면 API 낭비가 크다.
- 최종 후보를 판단할 때는 매우 유용하다.

### 13.4 브랜드 키트 초안

- 긴 소개
- 톤앤매너 키워드
- 금지 표현
- 로고 방향 키워드

이 기능은 향후 유료 기능 또는 확장 기능으로 좋다.

### 13.5 수동 체크리스트

무료 API 중심 MVP에서는 외부 도메인/상표 API를 바로 붙이지 않는다.

대신 최종 후보 옆에 체크리스트를 둔다.

- Google 검색
- 도메인 검색
- SNS 핸들 검색
- KIPRIS 상표 검색
- App Store / Play Store 검색

나중에 사용량이 생기면 도메인 검색 API, 상표 검색 보조 기능을 붙인다.

## 21. 사용자 답변 반영 사항

2026-06-21 기준 사용자 답변을 반영해 MVP 방향을 아래처럼 확정한다.

### 21.1 배포와 접근 방식

- 개인용 웹앱으로 시작한다.
- GitHub 저장소에서 작업하고 Vercel에 배포한다.
- 초기에는 계정 로그인 없이 비밀번호로 접속한다.
- 나중에 아이디/로그인 기능을 추가할 수 있게 인증 구조를 분리해둔다.

권장 방식:

- `APP_PASSWORD` 환경변수를 Vercel에 저장한다.
- 접속 시 비밀번호를 입력하면 서버가 세션 쿠키를 발급한다.
- Gemini API 키는 서버 환경변수 `GEMINI_API_KEY`로 관리한다.
- API 키는 브라우저에 노출하지 않는다.

### 21.2 언어 방향

- 기본 네이밍 방향은 영어 우선, 글로벌 지향이다.
- 한글/영어 혼합 또는 한글 단독도 가능하지만 글로벌 관점에서 자연스러운 경우에만 추천한다.
- 결과 카드에는 글로벌 발음성, 비영어권 오해 가능성, 영어권 자연스러움을 함께 표시한다.

### 21.3 저장 방식

- MVP 초기에는 로컬 저장 중심으로 빠르게 만든다.
- 여러 컴퓨터에서 사용할 예정이므로, Vercel 배포 후에는 “내보내기/가져오기”를 제공해 로컬 저장의 한계를 보완한다.
- 로그인 기능 추가 전까지 클라우드 동기화는 제외한다.

### 21.4 생성 제한

- 개인용이므로 하루 생성 횟수 제한은 두지 않는다.
- Gemini 무료 티어 보호는 사용을 방해하지 않는 완충장치 수준으로 둔다.
  - 한 번에 하나의 생성 요청만 허용
  - 생성 버튼은 기본 8초 재클릭 완충
  - 동일 입력값은 캐시 우선 사용
  - 429 rate limit 응답 시 Gemini가 허용할 때까지 잠시 대기하도록 안내
  - 재생성은 가능하되, 선택 후보 기반 변주를 더 눈에 띄게 제공

공식 문서 기준:

- Gemini API rate limits는 RPM, TPM, RPD 기준으로 적용된다.
- 실제 활성 한도는 모델과 프로젝트 상태에 따라 AI Studio에서 확인해야 한다.
- 무료 티어는 일부 모델 접근, 무료 입력/출력 토큰을 제공하지만 입력 콘텐츠가 제품 개선에 사용될 수 있다.

참고:

- https://ai.google.dev/gemini-api/docs/rate-limits
- https://ai.google.dev/gemini-api/docs/pricing

### 21.5 결과 수와 유료화

- 기본 생성 결과는 12개로 확정한다.
- 향후 유료화 가능성이 있으므로, MVP 단계에서도 기능 플래그 또는 플랜 분기 지점을 염두에 둔다.

유료화 후보:

- 클라우드 프로젝트 저장
- 브랜드 키트 생성
- 고급 후보 분석
- 도메인/SNS/상표 보조 체크
- 공유 링크
- 프로젝트 히스토리
